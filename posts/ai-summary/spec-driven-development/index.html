Spec-Driven Development：AI 编程时代的正确打开方式

10 Jun 2026
# 3447 字 ~ 约 9 分钟

从 Vibe Coding 到 Spec Coding

用 AI 写代码已经不是新鲜事。大多数人的起步方式都是类似的：脑中大概有个想法，直接丢给 AI 让它生成。这种方式被称为 Vibe Coding——凭感觉开发。

Vibe Coding 在原型验证阶段很好用：上手快、反馈快、适合小实验。但一旦需求变复杂，问题就接踵而至：

  • AI 漏掉关键边界条件
  • 理解偏差，写出来的东西方向不对
  • 架构风格前后不一致
  • 修改一处引发多处回归
  • 换个对话窗口，代码风格和实现策略全变了

这些问题的本质不是 AI 不会写代码，而是输入过于模糊。你只说了"帮我做个登录页面"，AI 只能猜测你到底想要什么。

这就引出了 Spec Coding 的核心理念：先把需求、约束、边界和验收标准写清楚，再让 AI 按规范实现。

方式对 AI 的定位主要特征
Vibe Coding即兴生成者靠感觉、靠上下文、靠运气
Spec Coding按施工图执行的工程队靠契约、靠边界、靠规范

两者的差别不在于是否使用 AI，而在于如何使用 AI。Vibe Coding 适合探索，Spec Coding 适合进入可交付、可维护、可复用的工程阶段。

什么是 Spec

Spec 是 Specification（规格/规范文档）的简称。在 AI 辅助编程场景中，Spec 可以理解为一份写给 AI 的结构化自然语言契约。

它的核心作用是把模糊想法转化为明确、可执行、可验证的开发指令。用一句话概括：Spec 是给 AI 写的"开发施工图 + 验收合同"。

一份好的 Spec 通常涵盖：

  • 对产品需求的定义
  • 对实现方式的约束
  • 对结果质量的验收标准
  • 对测试方式的明确说明

在 AI Coding 中，代码不应该是凭感觉生成的产物，而应该是 Spec 的衍生物。真正的源头不是代码，而是 Spec。 因此，Spec 也可以被视为 AI 编程时代的单一事实来源（Single Source of Truth）。

一份好的 Spec 应该包含什么

一份有用的 Spec 不一定很长，但必须清晰。通常至少包含以下六个部分。

需求目标：明确要做什么

不要只描述表面任务，要说明功能背后的业务意图：

  • 解决什么问题
  • 给谁使用
  • 触发场景是什么
  • 最终希望达成什么结果

关键是让 AI 明白"为什么做"，而不只是"做什么"。

行为要求：明确系统应该怎么做

描述系统在不同条件下的具体表现：

  • 在什么条件下触发
  • 系统应该执行什么动作
  • 异常情况如何处理
  • 哪些行为不允许发生

这部分越具体，AI 输出越稳定。AI 最怕的不是任务复杂，而是需求看起来清楚、实际缺少关键规则。

验收标准：明确做到什么程度才算完成

没有验收标准，AI 只能猜测你是否满意。验收标准应该可观察、可测试、可复现。例如：

  • 用户输入错误密码时，必须提示错误信息
  • 连续失败 5 次后，账号应被临时锁定
  • 登录成功后，必须跳转到指定页面
  • 所有接口响应时间应低于指定阈值

技术与架构约束：明确怎么做才不跑偏

防止"功能做对了，但实现方式不适合当前系统"的情况：

  • 必须使用的技术栈
  • 不允许引入的依赖
  • 需要遵守的目录结构和代码规范
  • 错误处理方式
  • 日志、权限、安全、性能要求

这一部分在团队开发和生产环境中尤其重要。

数据模型与边界条件

很多 bug 不是主流程错误，而是边界条件没有提前定义。需要写清楚：

  • 数据结构长什么样
  • 关键字段有哪些
  • 接口输入输出格式
  • 空值、异常值、重复值如何处理
  • 并发、权限、状态流转规则

测试策略：明确如何证明符合 Spec

不只是生成代码，而是生成可验证的结果：

  • 需要哪些单元测试
  • 需要哪些集成测试
  • 哪些关键路径必须验证
  • 验证成功的条件是什么

测试是 Spec 闭环的一部分。没有验证，Spec 只能停留在意图层面。

Spec-Driven Development 工作流

成熟的 Spec-Driven Development 包含四个步骤，形成一个完整的闭环。

第一步：定义 Spec。 整理目标、需求、边界和验收标准。如果只有粗略想法，可以先让 AI 帮忙扩写成规范草案。关键不是一次写完，而是先建立规则底座。

第二步：制定 Plan。 在 Spec 基础上补充技术约束和架构要求，让 AI 输出技术方案、模块拆解、实施顺序、风险点和任务清单。先想清楚"怎么做"，再进入实现。

第三步：按 Spec 实现。 AI 不再凭猜测生成代码，而是执行已经约定好的规范。由于目标、边界和约束已经清楚，代码产出的稳定性、一致性和可控性都会更高。

第四步：按 Spec 验证。 检查结果是否符合最初定义的规范——功能是否齐全、边界是否覆盖、测试是否通过、性能和安全要求是否满足、是否与架构约束一致。

这不是"生成了就算完成"，而是**“符合规范才算完成”**。

为什么 Spec 在 AI Coding 时代变得重要

AI 写代码的能力已经足够强，真正稀缺的逐渐变成了约束能力。

新的协作分工正在形成：人类负责定义目标、边界、标准和约束；AI 负责生成、补全、实现和重复性执行。人类更擅长抽象问题、明确目标、判断优先级、定义边界、制定标准；AI 更擅长快速生成、大量执行、代码补全、结构展开、测试铺设。

Spec 之所以重要，不是因为它更正式，而是因为它解决了 AI 编程时代最核心的问题：如何把强大的生成能力转化为稳定的交付能力。

实践建议

不要只给一句模糊需求。 至少补充目标用户、主要流程、成功标准、必须处理的情况、不能用的实现方式。哪怕只多写几行约束，AI 输出质量也会明显提升。

把验收标准单独写出来。 很多需求文档写了大量背景，却没有定义"什么叫完成"。一旦把验收标准写清楚，AI 就能围绕明确目标实现和自检。

把反复强调的约束沉淀成模板。 如果总是反复提醒 AI 使用某个技术栈、遵守某种目录结构、统一异常处理、不引入新依赖、必须补测试——这些内容就应该沉淀为长期复用的 Spec 模板。

复杂任务先让 AI 补全 Spec。 不要直接开写。先让 AI 检查需求是否有漏洞、边界条件是否遗漏、验收标准是否可测试、技术方案是否存在冲突。Spec 清楚后，实现阶段会更顺畅。

控制 Spec 的颗粒度。 大 Spec（300 行/70+ 任务）首次完成度仅约 60%，需要几十轮交互；小 Spec（77 行/10+ 任务）首次完成度可达 95%，3-5 轮追加即可完成。轻量化 Spec + 多轮交互往往更加高效。

写在最后

AI Coding 的关键能力正在从"会不会写 Prompt"转向"会不会写 Spec"。

Prompt 更偏向单次交互的表达技巧，Spec 则更像面向工程交付的约束系统。高质量 AI 编程不只是让 AI 写得快，而是先定义清楚：要做什么、不能做什么、做到什么程度算完成，以及如何验证结果。

人写 Spec，AI 写代码；写好 Spec，是 AI Coding 时代最重要的基本功。
