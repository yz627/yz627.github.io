Cursor Agent 体系全景汇总

11 Jun 2026
# 5884 字 ~ 约 15 分钟
  • 更新日期: 2026-06-09
  • 适用版本: Cursor 3.x
  • 参考：Cursor

总览

Cursor 的 Agent 体系不是单一功能，而是一套分层的能力架构。从最轻量的行为约束（Rules）到完全自主的云端自动化（Automations），覆盖了从"微调 AI 行为"到"AI 替你值守"的完整光谱。下表给出全局视图，后续逐一展开。

行为配置层（不写代码，只写文本）

Rules——持续生效的系统级约束

Rules 是注入到 Agent 上下文中的持久性指令，定义"AI 应该遵守什么规范"。它不提供新能力，只约束行为。

配置位置：.cursor/rules/ 目录，每个文件是一个 .mdc 规则文件。

触发机制：支持条件触发。可以在 .mdc 文件的 frontmatter 中指定 glob 模式（如 *.py、src/**/*.ts），只有当 Agent 操作匹配文件时才激活对应规则。也支持全局规则（始终生效）。

典型内容：编码规范（如"使用 Tailwind 而非内联样式"）、commit 规范（如"commit message 用中文，不超过 50 字"）、架构约束（如"所有 API 错误统一用 AppError 类"）。

示例：

---
globs: ["*.py"]
description: Python 代码规范
---
- 所有函数必须写 type hints
- 使用 logging 模块而非 print
- 异常处理统一使用自定义 AppError 类
- 测试文件放在 tests/ 目录，文件名以 test_ 开头

Skills——可复用的任务执行流程

Skills 是比 Rules 更"重"的能力模块。如果说 Rules 是"规矩"，Skills 就是"操作手册"——定义 AI 在特定场景下应该按什么流程执行。

配置位置：.agents/skills/ 目录（或 .cursor/skills/），每个 Skill 是一个文件夹，内含 SKILL.md。

触发机制：按需加载。Agent 根据用户意图自动判断是否需要激活某个 Skill，也可以通过 @skill-name 显式调用。

示例：

---
name: api-development
description: 后端 API 开发标准流程
triggers: ["创建API", "新增接口", "REST"]
---
# API 开发 Skill
## 步骤
1. 在 routes/ 下创建路由文件，遵循 RESTful 命名
2. 使用 zod 做请求参数校验
3. 错误响应统一用 { code, message } 格式
4. 在 tests/api/ 下编写集成测试

AGENTS.md——项目名片

AGENTS.md 是放在项目根目录的说明文件，相当于给 AI 的"项目简介"。

配置位置：项目根目录下的 AGENTS.md。

触发机制：始终加载，Agent 打开项目时自动读取。

典型内容：项目概览、技术栈、目录结构说明、核心模块介绍、常见工作流程。

交互式 Agent 层（人在回路中）

主 Agent——日常编码助手

主 Agent 是 Cursor 的核心，就是你在编辑器里直接对话的那个 AI。

触发方式：Cmd+L（打开聊天面板）或 Cmd+K（内联编辑）。

能力范围：读写文件、执行终端命令、搜索代码、多轮对话推理。它能使用所有已配置的 Rules、Skills、MCP Server 和内置工具。

工作模式：

  • Ask 模式：只回答，不修改代码
  • Agent 模式：自主规划、调用工具、修改代码（默认）
  • Plan 模式：先输出方案计划，确认后再执行

Subagents——专项子智能体

Subagents 是由主 Agent 创建和调度的专业化 Agent，用于处理需要独立上下文的子任务。

定义位置：.cursor/agents/ 目录，每个 Subagent 是一个 YAML 文件。

核心特性：

  • 拥有独立的上下文窗口，不与主 Agent 共享对话历史
  • 可以配置专属的 Prompt、模型、工具权限
  • 完成后将结果摘要回报给主 Agent

同步 vs 异步：

  • 同步 Subagent（Cursor 2.4+）：主 Agent 等待子任务完成后继续
  • 异步 Subagent（Cursor 2.5+）：主 Agent 无需等待，可以继续执行其他任务，子任务在后台并行运行

示例配置：

name: security-auditor
description: 专注安全审计的子智能体，只读分析不修改代码
model: claude-sonnet-4-20250514
tools:
  - read_file
  - search_files
  - run_terminal_command
prompt: |
  你是一个安全审计专家。只分析代码中的安全漏洞，不做任何修改。
  重点关注：SQL注入、XSS、敏感信息泄露、不安全的依赖。
  输出格式：漏洞列表 + 风险等级 + 修复建议。

Background Agent——后台异步 Agent

Background Agent 在 Cursor 1.0 中全员开放，是 Cloud Agent 的前身/轻量版。

触发方式：按 Cmd/Ctrl+E 或在 Agent 面板中选择"后台运行"。

核心特性：

  • 在后台异步执行任务，不阻塞当前编码
  • 运行在云端环境
  • 完成后通知用户审查结果

云端 Agent 层（脱离本地 IDE）

Cloud Agent——云端独立智能体

Cloud Agent 运行在 Cursor 提供的隔离云端虚拟机中，完全不依赖本地设备。

触发入口：

  • Cursor 编辑器内（Agent 面板）
  • Web 端（cursor.com/agents）
  • Slack（@cursor + 任务描述）
  • Linear / GitHub Issues

核心特性：

  • 完全云端运行，本地关机不受影响
  • 支持多任务并行，每个 Agent 在独立 VM 中
  • 自带完整开发环境（clone 仓库、安装依赖、运行构建和测试）
  • 可启动应用并截图验证 UI 变更

与 Background Agent 的区别：Background Agent 更偏"后台执行编辑器内的任务"，Cloud Agent 是"完全独立的云端开发者"，支持更长的运行时间、更多入口、更完整的开发环境。

Bugbot——PR 自动审查

Bugbot 是 Cursor 1.0 推出的代码审查 Agent，专门用于自动审查 GitHub PR。

触发方式：在 GitHub PR 中 @cursor 或配置自动触发。

核心特性：

  • 自动分析 PR 中的代码变更
  • 使用 Cursor 最强的模型进行审查
  • 在 PR 上留下审查评论（问题、建议、风险）

Automations——事件驱动自动化

Automations 是 Cursor 在 2026 年 3 月推出的功能，让 Agent 从"被动响应"变成"主动值守"。

触发方式：

  • GitHub 事件：Issue 创建、PR 提交、CI 失败等
  • Slack 事件：特定频道的消息、@cursor 提及
  • 定时任务：按计划周期性执行（如每天早上 9 点）
  • Webhook：外部系统触发

核心特性：

  • 基于规则（Rules）定义触发条件和行为
  • 内部调用 Cloud Agent 执行具体任务
  • 无需人工触发，7x24 小时运行

与 Cloud Agent 的关系：Automations 是"调度层"，Cloud Agent 是"执行层"。Automations 定义"什么时候做什么"，Cloud Agent 负责"具体怎么做"。

编程集成层（面向开发者）

Cursor SDK——编程式 Agent API

Cursor SDK 让开发者用 TypeScript 代码调用 Cursor 的 Agent 能力，将其嵌入到自己的工作流中。

安装方式：npm install @cursor/sdk

三种运行时：

  • Local：在本机运行，直接操作当前工作目录的文件，适合脚本、CI 检查
  • Cloud：在 Cursor 的云端 VM 运行，自动 clone 仓库、启动完整 dev 环境，适合独立任务
  • Self-hosted：在自己的服务器上运行，适合企业级部署

示例：

import { Agent } from "@cursor/sdk";

const agent = new Agent({
  runtime: "local",  // 或 "cloud"
  model: "claude-sonnet-4-20250514",
});

const result = await agent.prompt({
  message: "找到所有未处理的 TODO 注释并生成修复计划",
  contextDirs: ["./src"],
});

console.log(result.output);

外部集成层

MCP (Model Context Protocol)

MCP 是 Anthropic 提出的开放协议，Cursor 原生支持。它让 Agent 能调用外部工具和服务。

配置方式：~/.cursor/mcp.json 或通过 Cursor 的 MCP Marketplace 一键安装。

作用：让 Agent 获得"超能力"，如访问数据库、调用 Jira API、操作浏览器、查询内部系统等。

与 Tools 的关系：MCP Server 可以暴露多个 Tool，Agent 通过 MCP 协议远程调用它们，和本地内置 Tool 的使用方式一致。
