架构图渲染测试

11 Jun 2026
# 2381 字 ~ 约 6 分钟

目的：验证 render-codeblock 钩子对架构图的自动检测、显式 ```arch 标记、以及 arch shortcode 的三种触发方式是否均能统一渲染为深色主题图框。

1. 自动检测：方框字符

只要代码块中出现 ┌ ┐ └ ┘ 等方框字符，就会自动被识别为架构图，无需任何语言标签。

┌─────────────────────────────────────────────────────┐
│                   用户交互层                          │
│            (Web UI / CLI / IM Channels)              │
├─────────────────────────────────────────────────────┤
│                   Gateway 网关层                      │
│        (统一通信 / 适配器模式 / ChannelPlugin)         │
│        Bindings 路由 ── 按渠道分发消息到对应 Agent      │
│        Mappings  ── 消息路由到正确的 Agent             │
├─────────────────────────────────────────────────────┤
│                   Agent 核心层                        │
│   ┌───────────┐  ┌──────────┐  ┌──────────────┐    │
│   │ 规划引擎   │  │ 记忆系统  │  │  工具管理     │    │
│   └───────────┘  └──────────┘  └──────────────┘    │
└─────────────────────────────────────────────────────┘

2. 自动检测：流程箭头 + 连接符

包含 ▼ 等箭头并配合 ├ └ │ ─ 连接符的流程图也会被自动识别。

用户: "帮我分析这三个竞品的优劣势"
         │
         ▼
    主 Agent（判断：三个竞品互不依赖，可以并行）
         │
         ├── sessions_spawn(task="分析竞品A")  → SubAgent 1
         ├── sessions_spawn(task="分析竞品B")  → SubAgent 2
         └── sessions_spawn(task="分析竞品C")  → SubAgent 3
         │
         ▼  （三个 SubAgent 并行执行，非阻塞）
         │
    结果通过 Push 机制自动回传主 Agent

3. 显式 ```arch：纯箭头生命周期

只有 → 的简单流程图不含方框字符，需要显式标记为 arch。

创建(Created) → 执行(Running) → 等待(Waiting) → 完成(Completed) → 终止(Terminated)

4. 显式 ```flow：串行编排

flow 也是被识别的语言提示。

SubAgent A (需求分析) → SubAgent B (代码编写) → SubAgent C (代码审查) → SubAgent D (测试验证)

5. arch Shortcode + 标题

使用 shortcode 可以附加标题和主题变体。

组合模式：持久 Agent + 临时 SubAgent

                     Bindings 路由
GitHub PR 事件 ──────────────────→ coding-agent（长期驻留）
                                      │
                                      │ sessions_spawn
                                      ├→ SubAgent: 安全审查（临时）
                                      ├→ SubAgent: 性能分析（临时）
                                      └→ SubAgent: 风格检查（临时）

6. arch Shortcode + accent 主题

accent 主题变体

┌──────────┐    ┌──────────┐    ┌──────────┐
│  Channel │ →  │ Gateway  │ →  │  Agent   │
└──────────┘    └──────────┘    └──────────┘

7. 不应被误识别的普通代码块

7.1 目录树（无方框字符）

目录树使用 ├ └ │ 但没有 ┌ ┐ └ ┘ 等方框字符，应保持普通代码块样式。

~/.openclaw/
├── agents/                  # Agent 运行时数据（会话、持久化状态）
│   └── Subagent-name/       # 额外 Agent 运行时数据
├── cache/                   # 缓存目录
├── openclaw.json            # 主配置文件
├── plugins/                 # 插件安装状态
├── state/                   # OpenClaw 主状态数据库
└── workspace/               # Agent 工作空间

7.2 普通代码（无图形字符）

def fib(n):
    a, b = 0, 1
    for _ in range(n):
        yield a
        a, b = b, a + b

print(list(fib(10)))
GO main.go
package main

import "fmt"

func main() {
    fmt.Println("Hello, yzBlog!")
}

8. 对比表格

触发方式语法适用场景
自动检测普通 ``` 代码块含方框字符或 ▼+连接符 的标准架构图
显式语言标签```arch / ```flow / ```diagram纯箭头流程图、生命周期、串行编排
Shortcode{{< arch title="..." theme="..." >}}需要标题或特定主题变体时

9. 总结

三种触发方式最终渲染为统一的深色主题图框：等宽字体、圆角边框、柔和发光、响应式内边距。浅色模式下同样适配。后续新增架构图时无需任何额外配置，只要图中包含识别字符即可自动生效。
