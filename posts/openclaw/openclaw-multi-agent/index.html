OpenClaw 多 Agent 与 SubAgent 机制详解

11 Jun 2026
# 7902 字 ~ 约 20 分钟

整体架构概览

OpenClaw（原名 Clawdbot → Moltbot → OpenClaw）是一款遵循 MIT 协议的开源 AI Agent 平台。其 Agent 系统是整个平台的核心模块，包含约 686 个文件、13.5 万行代码。整体架构分为四个主要层次：

┌─────────────────────────────────────────────────────┐
│                   用户交互层                          │
│          (Web UI / CLI / IM Channels)               │
├─────────────────────────────────────────────────────┤
│                   Gateway 网关层                      │
│      (统一通信 / 适配器模式 / ChannelPlugin)           │
│      Bindings 路由 ── 按渠道分发消息到对应 Agent       │
│      Mappings  ── 消息路由到正确的 Agent               │
├─────────────────────────────────────────────────────┤
│                   Agent 核心层                        │
│   ┌───────────┐  ┌──────────┐  ┌──────────────┐    │
│   │ 规划引擎   │  │ 记忆系统  │  │  工具管理     │    │
│   └───────────┘  └──────────┘  └──────────────┘    │
│   ┌───────────┐  ┌──────────┐  ┌──────────────┐    │
│   │ SubAgent  │  │ Bindings │  │  Sessions    │    │
│   │  管理器    │  │  路由     │  │  会话隔离     │    │
│   └───────────┘  └──────────┘  └──────────────┘    │
├─────────────────────────────────────────────────────┤
│                   执行引擎层                          │
│   ┌───────────────┐  ┌─────────────────────┐       │
│   │ PI Embedded   │  │  ACP Protocol       │       │
│   │ Runner (内嵌)  │  │  Runner (跨进程)     │       │
│   └───────────────┘  └─────────────────────┘       │
│   ┌───────────────┐  ┌─────────────────────┐       │
│   │ Sandbox       │  │  工具过滤 Pipeline   │       │
│   │ (Docker隔离)  │  │  (7层策略管线)       │       │
│   └───────────────┘  └─────────────────────┘       │
└─────────────────────────────────────────────────────┘

其多 Agent 协作的设计哲学是"主从架构 + 声明式编排 + 消息路由"，核心理念为"让 AI 的错误可控，而非消除错误"。

两种多 Agent 模式：预先配置 vs 临时创建

OpenClaw 的多 Agent 体系存在两种截然不同的 Agent 类型，它们解决的是不同层次的问题。

预先配置的持久 Agent

在 openclaw.json 中静态定义一组长期驻留的 Agent，每个 Agent 拥有独立的身份、工作空间、记忆和 SOUL.md 人格定义：

{
  "agents": [
    {
      "id": "coding-agent",
      "name": "编码助手",
      "workspace": "./coding",
      "soul": "SOUL.md",
      "model": "claude-sonnet",
      "bindings": [
        { "channel": "github", "event": "pull_request" }
      ]
    },
    {
      "id": "support-agent",
      "name": "客服助手",
      "workspace": "./support",
      "soul": "SOUL.md",
      "model": "claude-haiku",
      "bindings": [
        { "channel": "slack", "pattern": "#support" }
      ]
    }
  ]
}

这些 Agent 通过 Bindings 路由机制接收消息。Bindings 本质上是一个路由表，根据消息来源的渠道、事件类型或关键词，将请求分发给对应的 Agent。配置支持热加载，修改后无需重启。

预先配置的 Agent 解决的是"谁负责什么领域“的问题——不同渠道的专属助手，各司其职，维护各自的长期记忆。

临时创建的 SubAgent

SubAgent 是主 Agent 在运行时通过 sessions_spawn 工具动态创建的临时并行单元，按需创建、用完即销。它解决的是”这个具体任务怎么并行拆解“的问题。

用户: "帮我分析这三个竞品的优劣势"
         │
         ▼
    主 Agent（判断：三个竞品互不依赖，可以并行）
         │
         ├── sessions_spawn(task="分析竞品A")  → SubAgent 1
         ├── sessions_spawn(task="分析竞品B")  → SubAgent 2
         └── sessions_spawn(task="分析竞品C")  → SubAgent 3
         │
         ▼  （三个 SubAgent 并行执行，非阻塞）
         │
    结果通过 Push 机制自动回传主 Agent
         │
         ▼
    主 Agent 汇总三份报告，统一交付
         │
         ▼
    三个 SubAgent 被终止、资源回收

核心对比

维度预先配置的持久 Agent临时创建的 SubAgent
创建方式openclaw.json 静态定义sessions_spawn 动态创建
生命周期长期驻留用完即销
上下文维护独立长期记忆每次启动都是干净环境
分工方式预先规划，按领域划分运行时由主 Agent 决定
消息路由Bindings 按渠道自动路由主 Agent 主动调度
通信方式sessions_send 平等通信Push 机制回传结果
典型场景不同渠道的专属助手一次性并行任务分解

两种模式叠加使用

实际项目中两种模式经常组合使用。一个预先配置的持久 Agent 收到复杂任务后，可以作为主 Agent 临时创建多个 SubAgent 执行子任务：

                     Bindings 路由
GitHub PR 事件 ──────────────────→ coding-agent（长期驻留）
                                      │
                                      │ sessions_spawn
                                      ├→ SubAgent: 安全审查（临时）
                                      ├→ SubAgent: 性能分析（临时）
                                      └→ SubAgent: 风格检查（临时）

预先配置的 Agent 之间也可以通过 sessions_send 进行平等的跨 Agent 通信（类似"同事协作”），而 SubAgent 与主 Agent 之间则是上下级的"任务派发"关系。

SubAgent 机制详解

核心价值

SubAgent 作为主 Agent 动态派生的临时并行单元，提供四个核心能力：

并行处理——多个 SubAgent 同时执行不同任务，大幅缩短总体执行时间。

上下文隔离——每个 SubAgent 拥有独立的会话和上下文，避免不同任务之间的信息污染。

资源优化——SubAgent 可以指定不同的模型执行任务，简单任务用更小更便宜的模型，复杂任务用更强的模型。

错误隔离——一个 SubAgent 的失败不影响其他 SubAgent，支持重试与错误隔离机制。

核心原语：sessions_spawn

sessions_spawn 是创建子代理的核心工具，支持三种调用方式：

sessions_spawn({
  task: "分析项目中的API接口设计",
  model: "claude-sonnet",
  timeout: 120,
  mode: "run",
  sessionId: "api-review-001"
})

生命周期

创建(Created) → 执行(Running) → 等待(Waiting) → 完成(Completed) → 终止(Terminated)

关键特性是 非阻塞执行：sessions_spawn 调用后立即返回，主 Agent 不会被阻塞。结果通过 Push-based 通告机制自动推送至主 Agent。

前置配置

SubAgent 的创建需要满足两个条件：

在 AGENTS.md 中设置允许嵌套的深度：

## SubAgent Settings
maxSpawnDepth: 2

在 TOOLS.md 中启用相关工具：

## Allowed Tools
- sessions_spawn
- sessions_send

管理命令

命令说明
clawhub subagents list查看所有活跃的 SubAgent
clawhub subagents kill <id>终止指定的 SubAgent
clawhub subagents log <id>查看 SubAgent 的执行日志

系统全局配置 maxConcurrent（默认值 8）控制最大并发 SubAgent 数量。

三种编排模式

并行执行（Parallel）

多个 SubAgent 同时处理互不依赖的子任务：

主 Agent
  ├── spawn → SubAgent A (查询北京天气)
  ├── spawn → SubAgent B (查询上海天气)
  └── spawn → SubAgent C (查询广州天气)
       │
       ▼
  barrier: wait_all  (等待所有完成)
       │
       ▼
  汇总结果

串行执行（Sequential）

任务按顺序依次执行，前一个的输出作为后一个的输入：

SubAgent A (需求分析) → SubAgent B (代码编写) → SubAgent C (代码审查) → SubAgent D (测试验证)

条件执行（Conditional）

根据中间结果动态决定后续执行路径：

SubAgent A (分析代码变更)
    │
    ▼
  条件判断
    ├── 变更涉及API → SubAgent B (API兼容性检查)
    ├── 变更涉及UI  → SubAgent C (UI回归测试)
    └── 仅文档变更  → 跳过测试，直接完成

确定性编排：Lobster YAML 工作流

为什么需要确定性编排

LLM 自主决定流程控制会导致"确定性缺失"——同样的输入可能产生不同的执行路径。对生产环境的关键业务流程，这种不确定性不可接受。

OpenClaw 社区推荐使用 Lobster YAML 工作流引擎，将流程控制权从 LLM 手中收回到开发者手中。

三层架构

协调者层（Orchestrator）——负责任务调度和流程控制，不包含业务逻辑。

专家子 Agent 层（Expert SubAgents）——每个 SubAgent 专注一个领域，拥有独立的工具和提示词。

工具技能层（Tool Skills）——原子化的工具操作，供 SubAgent 调用。

YAML 编排示例

name: code-review-pipeline
description: "自动化代码审查管道"

steps:
  - name: analyze
    tool: sessions_spawn
    input:
      task: "分析代码变更，识别影响范围"
      model: "claude-sonnet"

  - name: review-parallel
    parallel:
      - tool: sessions_spawn
        input:
          task: "安全性审查"
      - tool: sessions_spawn
        input:
          task: "性能审查"
      - tool: sessions_spawn
        input:
          task: "代码风格检查"
          model: "claude-haiku"
    barrier: wait_all

  - name: summary
    tool: sessions_spawn
    input:
      task: "汇总所有审查结果，生成最终报告"
    depends_on:
      - review-parallel

双轨运行时：Embedded 与 ACP

OpenClaw 的 Agent 系统支持两种运行时模式：

Embedded Runner（内嵌模式）——SubAgent 运行在主 Agent 进程内，共享资源，延迟低，适合轻量级任务。

ACP Protocol Runner（跨进程模式）——基于 Agent Communication Protocol 实现跨进程通信，SubAgent 运行在独立进程中，提供完全的进程级隔离，适合需要运行不受信任代码等高风险场景。

export type AgentConfig = {
  id: string;
  runtime: "embedded" | "acp";
  sandbox?: {
    enabled: boolean;
    type: "docker" | "none";
    image?: string;
  };
  tools: ToolPolicy[];
};

安全机制

OpenClaw 通过三层安全机制保障多 Agent 协作的可靠性：

七层工具策略管线（Tool Policy Pipeline）——控制 Agent 和 SubAgent 对工具的访问权限，每层独立配置允许或拒绝的工具列表，确保最小权限原则。

工具策略隔离——每个 SubAgent 可以拥有独立的工具策略。例如安全审查 SubAgent 可以访问代码仓库但不能执行代码，测试 SubAgent 可以执行代码但不能访问外部网络。

Sandbox 隔离——通过 Docker 容器实现执行隔离，每个 SubAgent 可以运行在独立容器中，拥有独立的文件系统和网络环境。

Agent 间通信机制

OpenClaw 提供了四种不同层次的 Agent 间通信机制，适用于不同的协作关系和场景。

sessions_send——平等通信

用于在已有的、长期运行的 Agent 之间传递消息。这是一种"同事之间对话"的平等关系——发送方发出消息后，接收方在自己的会话上下文中独立处理，结果不会自动回传。

sessions_send({
  sessionId: "developer-session-001",
  message: "API接口已更新，请同步修改前端调用"
})

sessions_spawn——上下级通信

主 Agent 创建 SubAgent 时通过 task 参数传递指令，SubAgent 完成后结果通过 Push-based 通告机制自动回传主 Agent。这是单向的"上级下发、下级汇报"模式。

steer——运行中干预

steer 用于对正在运行的 SubAgent 进行实时方向调整，相当于"中途补充需求"。它不会终止 SubAgent 当前的工作，而是在其上下文中注入新的指导信息。

steer({
  sessionId: "subagent-001",
  instruction: "重点关注SQL注入风险，其他安全问题可以忽略"
})

全局 Memory——间接协同

多个 Agent 可以读写共享的记忆空间来协同状态，类似于"共享白板"模式——一个 Agent 写入分析结果，另一个 Agent 读取后继续处理。

通信方式选择指南

通信方式协作关系典型场景
sessions_send平等协作（同事）持久 Agent 之间交换信息
sessions_spawn上下级派发（领导-下属）任务分解、并行处理
steer运行中干预（补充指令）动态调整 SubAgent 执行方向
全局 Memory间接协同（共享白板）松耦合状态同步、数据传递

最佳实践

合理设置并发上限——maxConcurrent 默认值为 8，根据系统资源调整，避免过载。

设置合理的超时时间——为每个 SubAgent 设置 timeout，避免任务挂起导致资源泄漏。

明确任务边界——每个 SubAgent 的任务描述应该清晰、具体、自包含，避免歧义和上下文依赖。

限制嵌套深度——maxSpawnDepth 建议设置为 1-2，避免过深嵌套导致调试困难。

关键流程使用确定性编排——生产环境的关键业务流程使用 Lobster YAML 而非 LLM 自主编排。

工具策略隔离——为不同职责的 Agent/SubAgent 配置不同的工具访问权限，遵循最小权限原则。

总结

OpenClaw 的多 Agent 体系围绕两个核心维度展开：

第一个维度是预先配置的持久 Agent，通过 openclaw.json 静态定义、Bindings 路由消息、sessions_send 平等通信，解决"按领域分工"的问题。每个 Agent 长期驻留、维护独立记忆，适合不同渠道和领域的专属助手场景。

第二个维度是临时创建的 SubAgent，通过 sessions_spawn 动态创建、用完即销，解决"具体任务并行拆解"的问题。SubAgent 按需创建、上下文隔离、错误隔离，适合一次性并行任务处理。

两者可以灵活叠加——持久 Agent 作为主 Agent 接收任务后，动态创建 SubAgent 分解执行，构建出从简单的消息路由到复杂确定性工作流编排的完整能力栈。其设计哲学是在保持 LLM 灵活性的同时，通过声明式配置、隔离沙箱和确定性编排来约束和控制 AI 的行为边界，使多 Agent 协作真正可用于生产环境。
